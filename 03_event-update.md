# 03_event-update.md  
Thesis-Narrative OS — Event Update Specification

## 1. Definition of an Event Update

An Event Update is defined as the moment when a character’s  
**Value Vector undergoes a change**.

Within the OS, an event is established at the intersection of:

- **External change** (recorded in the Fact Log)  
- **Internal change** (Value Vector Update)  
- **Change in distance/direction relative to the Thesis (ΔValueVector)**  

An event is not a “story incident” in the traditional sense.  
It is a **computational fact**:  
a measurable update in the value‑vector space.

---

## 2. Events Are Structurally Determined by the Thesis and the Protagonist

In the Thesis-Narrative OS, events are not “invented” by the creator.  
They are **structurally implied update points** derived from:

- the Thesis  
- the Thesis Type (Question / Assertion)  
- the Protagonist’s Value Vector  

### 2.1 The Thesis is defined  
The Thesis provides the **directional vector** that anchors the narrative.

### 2.2 The Thesis Type is defined (Question / Assertion)  
- **Question Type** → the protagonist tends to *oscillate*  
- **Assertion Type** → the protagonist tends to *impact* or *shake* the Thesis  

This determines the **shape of the narrative trajectory**.

### 2.3 The Protagonist is determined  
The protagonist is the character whose Value Vector:

**produces the greatest structural change or influence  
in relation to the Thesis.**

This influence may take two forms:

- **Internal oscillation**  
  The character’s own Value Vector fluctuates significantly.

- **External impact**  
  The character *shakes or redefines* the meaning or position of the Thesis itself.

The protagonist is therefore the **central update node**  
within the Thesis–Character relationship.

### 2.4 Event Updates arise from the difference between the Thesis and the Protagonist  
By comparing the protagonist’s Value Vector with the Thesis vector,  
the OS identifies **necessary update points (milestones)**.

Formally:



\[
Event = f(\Delta ValueVector)
\]



Thus, events are not arbitrary.  
They are **structurally required updates** within the system.

---

## 3. Structure of an Event Update

An Event Update consists of three layers:

### 3.1 External Layer  
Changes in the external world, recorded in the Fact Log.

### 3.2 Internal Layer  
The Value Vector update (ΔValueVector).

### 3.3 Thesis Layer  
Interpretation of the update relative to the Thesis:

- convergence  
- divergence  
- oscillation  
- impact (shaking the Thesis)

Only when all three layers are present does an event exist within the OS.

---

## 4. Structural Types of Event Updates

Event Updates are classified by the direction of the update:

### 4.1 Convergence Event  
The Value Vector moves toward the Thesis.

### 4.2 Divergence Event  
The Value Vector moves away from the Thesis.

### 4.3 Oscillation Event  
The Value Vector fluctuates with no stable direction.

### 4.4 Impact Event  
The update does not primarily change the protagonist,  
but instead **shakes or redefines the Thesis itself**.  
(Common in Assertion‑type narratives.)

These are structural categories,  
not narrative tropes or plot devices.

---

## 5. Determining the Event Sequence

The event sequence consists of:

- **Milestones (structurally required updates)**  
- **Paths (creator‑defined transitions)**

### 5.1 Milestones (Required Update Points)  
Milestones are determined automatically from:

- the Thesis  
- the protagonist’s Value Vector  
- the difference between them  

These define:

- required update directions  
- required update magnitudes  
- partially constrained ordering  

Milestones are **fixed by the OS**.

### 5.2 Paths (Free Components)  
Paths describe how the creator chooses to connect milestones:

- which external facts to use  
- which characters to involve  
- how to render the event  
- what sequence to present  

The OS defines **what must update**,  
not **how it is dramatized**.

---

## 6. Structural Requirements for an Event Update

An event must satisfy all of the following:

- A Value Vector is updated  
- The update direction is comparable to the Thesis  
- The external change is representable in the Fact Log  
- Subjective expression can be generated  
- World consistency is maintained  

If any of these conditions are missing,  
the OS does not treat the moment as an event.

---

## 7. Out‑of‑Scope (Non‑Structural Elements)

This chapter defines **only the structural aspects** of Event Updates.  
The following are intentionally excluded:

- the narrative content of events  
- how to dramatize or stage an event  
- which characters participate  
- how to invent events  
- plot construction techniques  
- story templates or formulas  

These depend on the creator’s internal model  
and are not part of the OS specification.
