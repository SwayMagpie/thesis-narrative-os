# 00_overview.md
Thesis-Narrative OS — Theory Overview

## 0. Conceptual Lineage — Origin of Value Dimensions

Thesis‑Narrative OS は、**VBPM-Core（Value-Based Personality Model）** に由来する  
「価値基準（Value Dimensions）」の概念を継承している。

VBPM-Core では、人格を  
**価値基準 × 重み × 最適化**  
で定義するために、行動や状況を評価する軸として  
**Value Dimensions（価値基準／価値次元）** が導入された。

この価値次元は、探索（exploration）や意思決定を成立させるために  
必須の内部評価構造であり、外部化可能なベクトルとして扱われる。

Thesis‑Narrative OS は、この価値次元の概念を  
**キャラクターの内部状態（Value Vector）** として再構成し、  
物語生成における価値観の揺れ・情動・主観表現・イベント更新の  
中心構造として利用する。

本 OS における「価値次元」は、  
一般的な創作論の“価値観”とは異なり、  
**計算可能・更新可能・外部化可能な内部評価軸**として定義される。

---

## 1. Core Insight

本 OS の発端は以下の内部計算から始まった：

- 「事実ログを束ねれば物語は作れる」
- しかし本質は“事実ログだけではない”
- **価値観 × イベント × 情動 × 主観表現 × 世界整合**  
  を OS 的に結合することで、物語生成は完全に形式化できる

本理論は、物語を  
**価値関数の最適化・状態遷移ログのレンダリング**  
として扱う。

---

## 2. 全体フロー（OS 的プロセス）

### 1. Thesis（テーゼ）を立てる  
物語の価値関数の中心。  
OS 的には「最終的に到達する価値次元の方向ベクトル」。

例：自由とは何か／孤独は悪か／正しさは誰が決めるのか

---

### 2. テーゼを揺らす価値観を配置する  
物語を動かすキャラクターの生成。

- テーゼを体現する価値観  
- テーゼを揺らす価値観  
- テーゼに揺れる価値観  

OS 的対応：

- Character = 価値次元の重みベクトル  
- Conflict = ベクトル差分  
- Narrative = 差分が時間軸で収束していく過程  

---

### 3. テーゼの伝え方（問い or 主張）を決める  
物語の構造フォーマットを決定。

- 問い型 → 主人公が探索する構造  
- 主張型 → 主人公が証明する構造  

OS 的には「価値関数の最適化経路の形状」を決める工程。

---

### 4. Event（価値観更新ポイント）を設定する  
イベント = 主人公の価値ベクトルが更新される瞬間  
＝ 主観世界の差分が最大化される瞬間

---

## 3. Emotion Dynamics（情動力学）— OS の中核

### 情動は「キャラクターの内から湧き上がる内部現象」  
外部刺激ではなく、  
**価値観（Value Vector）とイベント（Event）の摩擦**  
から自然発生する。

### 定式化  


\[
Emotion = f(\Delta ValueVector \times EventIntensity)
\]



- ΔValueVector = 価値観の揺れ  
- EventIntensity = イベントの強度  
- EmotionStrength = 情動の強さ  

### Emotion Purity（情動純度）  
物語の「快」

- 揺れの大きさ  
- 揺れの方向の明確さ  
- 外部要因に汚染されていない内部発生性  



\[
Purity = |\Delta ValueVector|
\]



情動純度は **価値観の揺れの大きさとして観測される**。

---

- 事実ログ = 外界の変化

主観表現は「キャラクターが世界をどう経験したか」のログ。

物語 = **価値観の変遷ログのレンダリング結果**

# 01_thesis-model.md  
Thesis-Narrative OS — Thesis Model Specification

## 1. Thesis（テーゼ）の定義

テーゼとは、物語全体を貫く **中心的な価値関数（Value Function）** である。  
キャラクターの価値ベクトル、イベント更新、情動生成、主観表現など、  
OS 内のすべての計算はテーゼを基準として構造化される。

テーゼは以下の性質を持つ：

- **方向性（Directionality）**  
  価値次元空間における「向かうべき方向ベクトル」を定義する。

- **一貫性（Consistency）**  
  物語全体を通して保持される中心軸となる。

- **更新可能性（Updatability）**  
  キャラクターの価値ベクトルがテーゼに収束・反発する形で変化する。

- **評価基準（Evaluation Basis）**  
  イベントや主観表現の意味づけに影響を与える。

---

## 2. Thesis の役割

Thesis は OS 内で以下の役割を担う：

### 2.1 Narrative Trajectory（物語軌道）の決定  
物語がどの方向へ進むかを決定する。  
キャラクターの価値ベクトルは、テーゼに対して：

- 近づく  
- 離れる  
- 揺れる  

といった形で変化し、これが物語の軌道を形成する。

### 2.2 Conflict Structure（対立構造）の生成  
テーゼとキャラクターの価値ベクトルの差分が  
**対立（Conflict）** を生む。

### 2.3 Event Update の基準  
イベントは「価値ベクトルが更新される瞬間」であり、  
その更新方向はテーゼとの関係によって決まる。

### 2.4 Emotion Dynamics への影響  
価値ベクトルの揺れ（ΔValueVector）は、  
テーゼとの距離・方向によって意味づけられ、  
情動強度・純度の計算に影響する。

---

## 3. Thesis の型（Structural Types）

Thesis は OS 内で以下の構造型に分類される。  
これは「テーゼの立て方」ではなく、  
**テーゼが OS 内でどのように機能するか**を示す分類である。

### 3.1 Question Type（問い型）  
- 中心的な問いを設定する構造  
- キャラクターは問いに対して探索的に動く  
- Narrative Trajectory は「探索曲線」になる

例：  
- 自由とは何か  
- 正しさは誰が決めるのか

### 3.2 Assertion Type（主張型）  
- テーゼが明確な主張として存在する  
- キャラクターはその主張を証明・反証する方向に動く  
- Narrative Trajectory は「証明曲線」になる

例：  
- 孤独は悪ではない  
- 正義は個人の選択に宿る

### 3.3 Directional Type（方向型）  
- 抽象的な価値方向を示す  
- キャラクターはその方向に収束・逸脱する形で動く  
- Narrative Trajectory は「収束曲線」になる

例：  
- 自律へ向かう  
- 調和へ向かう

---

## 4. Thesis と Value Dimensions の関係

Thesis は **価値次元空間における方向ベクトル** として定義される。  
これにより、以下の OS 内計算が可能になる：

- キャラクターの価値ベクトルとの距離計算  
- 価値ベクトル更新方向の決定  
- 情動純度の意味づけ  
- 主観表現の方向性の決定  

Thesis は「価値次元のどの方向が重要か」を示す  
**重み付けの基準点**として機能する。

---

## 5. Thesis と Event の関係

イベントは「価値ベクトルが更新される瞬間」であり、  
その更新方向は Thesis によって決まる。

- テーゼに近づく更新  
- テーゼから離れる更新  
- テーゼに対して揺れる更新  

これらの差分が Narrative Trajectory を形成する。

---

## 6. Thesis と Emotion Dynamics の関係

Emotion Dynamics の計算式：



\[
Emotion = f(\Delta ValueVector \times EventIntensity)
\]



ここで ΔValueVector の意味づけは Thesis に依存する。

- テーゼに近づく揺れ  
- テーゼから離れる揺れ  
- テーゼに対して方向が曖昧な揺れ  

これらが情動強度・純度の解釈に影響する。

---

## 7. Thesis の構造的要件（Structural Requirements）

Thesis は以下の要件を満たす必要がある：

- **方向性があること**  
  価値次元空間でベクトルとして扱える。

- **一貫性があること**  
  物語全体を通して中心軸として機能する。

- **抽象度が適切であること**  
  価値ベクトルの更新を許容するレベルの抽象性を持つ。

- **更新可能性があること**  
  キャラクターの変化を説明できる。

- **価値次元と整合すること**  
  価値ベクトルとの距離・方向が計算可能である。

---

## 8. 本章が提供しないもの（Non-Scope）

本章は **OS の構造のみ**を扱う。  
以下は本章の対象外であり、外部化しない：

- テーゼをどう思いつくか  
- テーゼをどう選ぶか  
- テーゼをどう評価するか  
- テーゼ生成の思考プロセス  
- 個人固有の価値観・感性  
- 内部 OS（主観世界モデル）の詳細  

これらは利用者の内部モデルに依存するため、  
OS の仕様書には含めない。


# 02_value-vector.md  
Thesis-Narrative OS — Value Vector Specification

## 1. Value Vector の定義

Value Vector は、キャラクターの **内部状態を表す多次元ベクトル** である。  
各次元は Value Dimension に対応し、キャラクターがどの方向へ揺れ、  
どの方向へ更新されるかを定義する。

Value Vector は以下の役割を持つ：

- テーゼとの距離・方向の計算  
- イベント更新の対象  
- 情動生成の基礎データ  
- 主観表現の源泉  

キャラクターとは、OS 的には **Value Vector の集合体** である。

---

## 2. キャラクター配置の基本原理

テーゼが決まった時点で、  
キャラクター配置は **「テーゼに対してどう揺れるか」** の問題に還元される。

OS 的には、キャラクターは以下の 2 系統に分類される：

- **テーゼを揺らすもの（回答を持つ者）**  
- **テーゼに揺れるもの（回答を持たない者）**

この分類はキャラクターの性格や役割ではなく、  
**価値ベクトルの構造的分類**である。

---

## 3. テーゼを揺らすもの（回答を持つ者）

テーゼに対して **自分なりの“回答”を持つキャラクター**。  
その回答が肯定か反発かは問わない。

### 3.1 テーゼの肯定者・体現者  
- テーゼと同方向の価値ベクトルを持つ  
- テーゼの方向性を強化する  
- Narrative Trajectory の「収束側」を形成する

### 3.2 テーゼの反発者  
- テーゼと逆方向の価値ベクトルを持つ  
- テーゼを揺らし、対立構造を生む  
- Narrative Trajectory の「反発側」を形成する

これらは OS 内で **テーゼの揺れ幅を作るための構造要素** である。

---

## 4. テーゼに揺れるもの（回答を持たない者）

テーゼに対して **自分の“回答”を持たないキャラクター**。  
価値ベクトルが未確定、または方向性が曖昧な状態にある。

特徴：

- テーゼの方向性に対して揺れ動く  
- イベントによって価値ベクトルが大きく更新される  
- Narrative Trajectory の「変化の中心」になりやすい

OS 的には、  
**ΔValueVector（揺れ）が最も大きく発生する構造要素** である。

---

## 5. キャラクター数の要件

Thesis-Narrative OS は、  
**キャラクター数に依存しない構造** を持つ。

極論として：

- **1キャラクターのみ**  
- 他はすべて **イベントとして外部化**

という構成でも物語は成立する。

理由：

- 価値ベクトルの更新  
- テーゼとの距離変化  
- 情動生成  
- 主観表現  
- 世界整合  

これらが OS 内で完結しているため、  
キャラクター数は Narrative Rendering の問題ではなく、  
**Value Vector の更新問題**に還元されるからである。

---

## 6. 本章が提供しないもの（Non-Scope）

本章は **Value Vector の構造のみ** を扱う。  
以下は対象外とする：

- キャラクター性格の作り方  
- キャラクターの背景設定  
- キャラクターの感情生成プロセス  
- キャラクターの思想形成  
- キャラクターの“回答”をどう思いつくか  
- キャラクターの創作手法全般  

これらは利用者の内部モデルに依存するため、  
OS の仕様書には含めない。


# 03_event-update.md  
Thesis-Narrative OS — Event Update Specification

## 1. Event Update の定義

Event Update とは、  
キャラクターの **Value Vector が更新される瞬間** を指す。

OS 内では、イベントは以下の 3 要素の交点として成立する：

- **外部状態の変化（Fact Log）**  
- **内部状態の変化（Value Vector Update）**  
- **テーゼとの距離・方向の変化（ΔValueVector）**

イベントは「出来事」ではなく、  
**価値ベクトルが変化したという計算上の事実**である。

---

## 2. イベントはテーゼと主人公から構造的に決まる

Thesis-Narrative OS では、  
イベントは創作者が“考える”ものではなく、  
**テーゼと主人公の関係から必然的に導かれる更新点**である。

### 2.1 テーゼが決まる  
テーゼは物語の中心となる **方向ベクトル** を定義する。

### 2.2 Thesis Type（Question / Assertion）が決まる  
- Question Type → 主人公が揺れやすい構造  
- Assertion Type → 主人公が揺らしやすい構造  

これにより、Narrative Trajectory の **更新軌道の形** が決まる。

### 2.3 主人公が決まる  
主人公とは、  
**テーゼとの関係において、物語上もっとも大きな変化・影響を担う Value Vector**  
を持つキャラクターである。

この「変化・影響」は次のいずれの形でもよい：

- **自身の Value Vector が大きく揺れる**（テーゼに揺れる）  
- **テーゼ側の意味づけや位置づけを揺らす**（テーゼを揺らす）

主人公は、テーゼとの関係における  
**主要な更新の中心** として Narrative Trajectory を形成する。

### 2.4 主人公とテーゼの差分からイベントが決まる  
主人公の Value Vector とテーゼの方向ベクトルの差分を取ると、  
**必要な更新点（マイルストーン）が自動的に決まる**。

OS 的には：



\[
Event = f(\Delta ValueVector)
\]



つまり、  
**イベントは“必要な更新点”として構造的に存在する**。

---

## 3. Event Update の構造

Event Update は以下の 3 層で構成される：

### 3.1 External Layer（外部変化）  
Fact Log によって記述される外部状態の変化。

### 3.2 Internal Layer（内部変化）  
Value Vector の更新量（ΔValueVector）。

### 3.3 Thesis Layer（意味づけ）  
更新がテーゼに対して：

- 近づく  
- 離れる  
- 揺れる  
- 揺らす  

のどれに該当するかを決定する。

---

## 4. Event Update の種類（構造分類）

イベントは、更新方向によって以下の 4 種に分類される：

### 4.1 Convergence Event（収束イベント）  
Value Vector がテーゼ方向に更新される。

### 4.2 Divergence Event（乖離イベント）  
Value Vector がテーゼから離れる方向に更新される。

### 4.3 Oscillation Event（揺れイベント）  
方向が定まらず、揺れ幅が大きい更新。

### 4.4 Impact Event（揺らしイベント）  
主人公側の更新ではなく、  
**テーゼ側の意味づけが揺らされる更新**。

（Assertion 型で特に発生しやすい）

---

## 5. Event Sequence（イベント列）の決定

イベント列は **マイルストーン（必然）** と **経路（自由）** に分かれる。

### 5.1 マイルストーン（必然的に必要な更新点）  
主人公とテーゼの差分から自動的に決まる。

- 更新すべき方向  
- 更新すべき量  
- 更新すべき順序（部分的）

これらは OS 内で構造的に固定される。

### 5.2 経路（自由に選べる部分）  
マイルストーン間をどう繋ぐかは自由。

- どの Fact Log を使うか  
- どのキャラクターを関与させるか  
- どのような演出にするか  

OS が規定するのは **更新点** であり、  
**演出や順番は OS の外側**である。

---

## 6. Event Update の要件（Structural Requirements）

イベントは以下の要件を満たす必要がある：

- Value Vector が更新されている  
- 更新方向がテーゼと比較可能である  
- Fact Log によって外部状態が説明可能である  
- 主観表現が生成可能である  
- 世界整合性を保てる  

これらを満たさないものは OS 内では「イベント」として扱われない。

---

## 7. 本章が提供しないもの（Non-Scope）

本章は Event Update の **構造のみ** を扱う。  
以下は対象外とする：

- イベントの内容（出来事）  
- どのような演出にするか  
- どのキャラクターを関与させるか  
- どのようにイベントを思いつくか  
- プロットの作り方  
- 物語のテンプレート  

これらは創作者の内部モデルに依存するため、  
OS の仕様書には含めない。
